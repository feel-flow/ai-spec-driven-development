# RISKS.md - リスク管理

## 1. リスク管理概要

### リスク管理方針
- **目的**: プロジェクトの成功を脅かす潜在的な問題を特定し、事前に対策を講じる
- **スコープ**: 技術、ビジネス、運用、セキュリティ、コンプライアンスのすべての側面
- **更新頻度**: 隔週（リスク評価会議にて）
- **責任者**: プロジェクトマネージャー

### リスクレベル定義
| レベル | 影響度 × 発生確率 | 対応 |
|---|---|---|
| 🔴 Critical | 9-10 | 即座に対応必須 |
| 🟠 High | 6-8 | 優先的に対策実施 |
| 🟡 Medium | 3-5 | 監視と計画的対応 |
| 🟢 Low | 1-2 | 定期的な監視 |

## 4. リスク対応戦略

### 対応戦略の種類
| 戦略 | 説明 | 適用例 |
|---|---|---|
| 回避 (Avoid) | リスクを完全に排除 | 危険な技術の不採用 |
| 転嫁 (Transfer) | リスクを第三者に移転 | 保険加入、外注 |
| 軽減 (Mitigate) | 影響や確率を低減 | バックアップ、冗長化 |
| 受容 (Accept) | リスクを受け入れる | 軽微な問題 |

## 6. コンティンジェンシープラン

### シナリオ別対応計画

#### シナリオ1: 本番環境の完全停止
**トリガー**: システム全体のダウンタイム発生
**対応手順**:
1. インシデント対策本部の設置（5分以内）
2. 影響範囲の特定と初期対応（15分以内）
3. ステークホルダーへの第一報（30分以内）
4. 復旧作業の実施
5. 原因分析と再発防止策の策定

#### シナリオ2: データ漏洩インシデント
**トリガー**: 個人情報を含むデータの不正アクセス検知
**対応手順**:
1. 該当システムの即時遮断
2. セキュリティチームへの通知
3. 影響範囲の特定
4. 法務・コンプライアンスチームとの連携
5. 必要に応じて当局への報告
6. 影響を受けた顧客への通知

#### シナリオ3: 主要ベンダーのサービス停止
**トリガー**: 依存する外部サービスの長時間停止
**対応手順**:
1. 代替サービスへの切り替え
2. キャッシュデータでの限定的運用
3. ユーザーへの通知と機能制限の案内
4. ベンダーとの連携と復旧見込みの確認
