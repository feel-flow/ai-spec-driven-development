## 2. Phase 1: Issue-First Development

### 2.1 原則

> **コードを書く前に、必ずIssueを作成する**

```
❌ Bad:  アイデア → コーディング → あとでIssue作成
✅ Good: アイデア → Issue作成 → 設計 → コーディング
```

### 2.2 Issue作成のベストプラクティス

#### Issue構成要素

```markdown
## 概要
[1-2文で何を実現するか]

## 背景・目的
[なぜこのIssueが必要か]

## 要件
### 機能要件
- [ ] 要件1
- [ ] 要件2

### 非機能要件
- [ ] パフォーマンス要件
- [ ] セキュリティ要件

## 技術仕様
[実装に必要な技術詳細]

## 受け入れ条件
- [ ] 条件1が満たされている
- [ ] テストが書かれている

## 依存関係
- depends on #XX
- blocks #YY

## 参考資料
- [関連ドキュメント](link)
```

#### Issue分類（ラベル）

| ラベル | 用途 |
|--------|------|
| `feature` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメント |
| `chore` | 雑務・設定 |
| `design` | 設計作業 |
| `database` | DB関連 |
| `api` | API関連 |
| `frontend` | フロントエンド |
| `backend` | バックエンド |

### 2.3 依存関係の管理

```
┌─────────────────────────────────────────────────────────────────┐
│                    Issue依存関係の例                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  #1 プロジェクト初期化                                           │
│       ↓                                                         │
│  #2 DBスキーマ設計  ←────────────────────┐                      │
│       ↓                                  │                      │
│  ┌────┴────┐                             │                      │
│  ↓         ↓                             │                      │
│  #3 認証    #4 ユーザー管理               │                      │
│       ↓         ↓                        │                      │
│       └────┬────┘                        │                      │
│            ↓                             │                      │
│       #5 コア機能A                        │                      │
│            ↓                             │                      │
│       #6 コア機能B  ─────────────────────┘                      │
│                    (DBスキーマに依存)                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4 典型的なIssue構成

新規プロジェクトで必要となる典型的なIssue:

| 優先度 | Issue | 依存関係 |
|-------|-------|----------|
| 1 | プロジェクト初期化・環境構築 | なし |
| 2 | DBスキーマ設計 | #1 |
| 3 | 認証・セッション管理 | #2 |
| 4 | ユーザー管理API | #2, #3 |
| 5 | コア機能実装 | #4 |
| 6 | 監査ログ・コンプライアンス | #2 |

## 3. Phase 2: Document-Driven Design

### 3.1 原則

> **技術決定は、ドキュメントに書いて確定させる**

```
❌ Bad:  口頭で合意 → 実装時に忘れる → 手戻り
✅ Good: ドキュメントに明記 → レビュー → 合意 → 実装
```

### 3.2 ARCHITECTURE.mdの構成

```markdown
# ARCHITECTURE.md

## 1. システム概要
- プロジェクトの目的
- 技術スタック
- アーキテクチャ図

## 2. データベース設計
- ER図
- テーブル定義（全カラム）
- インデックス設計
- 制約条件

## 3. API設計
- エンドポイント一覧
- リクエスト/レスポンス形式
- 認証・認可

## 4. 認証・セキュリティ
- 認証フロー
- JWT/セッション設計
- RBAC設計

## 5. 外部連携
- 連携サービス
- Webhook設計

## 6. 監視・運用
- ログ設計
- メトリクス
- アラート

## Revision History
- 変更履歴を記録
```

### 3.3 データベース設計の確定プロセス

```
┌─────────────────────────────────────────────────────────────────┐
│               DB設計確定プロセス                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 初期設計                                                     │
│     ├─ 主要エンティティの洗い出し                                 │
│     ├─ リレーション定義                                          │
│     └─ ARCHITECTURE.mdに記載                                    │
│                                                                 │
│  2. MECE検証（後述）                                             │
│     ├─ 機能視点での検証                                          │
│     ├─ データフロー視点での検証                                   │
│     └─ 漏れているテーブルの追加                                   │
│                                                                 │
│  3. Issue更新                                                    │
│     ├─ DBスキーマIssueに反映                                     │
│     └─ 依存Issueに影響を反映                                     │
│                                                                 │
│  4. 最終確定                                                     │
│     ├─ Revision History更新                                     │
│     └─ マイグレーションファイル計画                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 設計レビューで発見される典型的な漏れ

| カテゴリ | 見落としやすいテーブル/機能 | 発見のきっかけ |
|---------|-------------------------|--------------|
| ユーザー管理 | プロフィール拡張情報 | ユーザー管理Issueのレビュー |
| 監査・ログ | 監査ログテーブル | コンプライアンス要件の確認 |
| 課金・契約 | 請求書・支払い履歴 | 契約管理の詳細化 |
| 通知 | 通知設定・履歴 | ユーザー体験の詳細化 |

## 4. Phase 3: MECE検証

### 4.1 原則

> **複数の視点から、漏れなくダブりなく検証する**

```
MECE = Mutually Exclusive, Collectively Exhaustive
     = 相互に排他的、全体として網羅的
```

### 4.2 検証の視点

```
┌─────────────────────────────────────────────────────────────────┐
│                    MECE検証の5つの視点                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────┐                                              │
│  │ 1. 機能視点    │  各機能に必要なデータは揃っているか？         │
│  └───────────────┘                                              │
│         ↓                                                       │
│  ┌───────────────┐                                              │
│  │ 2. データフロー │  データの流れに抜けはないか？                │
│  │    視点       │  入力→処理→出力→保存                         │
│  └───────────────┘                                              │
│         ↓                                                       │
│  ┌───────────────┐                                              │
│  │ 3. ライフサイクル│ 作成→更新→削除→アーカイブは考慮済みか？    │
│  │    視点       │                                              │
│  └───────────────┘                                              │
│         ↓                                                       │
│  ┌───────────────┐                                              │
│  │ 4. 関係者視点  │  全てのアクター（ユーザー種別）を網羅？       │
│  └───────────────┘                                              │
│         ↓                                                       │
│  ┌───────────────┐                                              │
│  │ 5. 非機能視点  │  監査・ログ・メトリクス・通知は？            │
│  └───────────────┘                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 検証チェックリスト

#### データベース設計のMECE検証

```markdown
## 機能視点
- [ ] 認証に必要なテーブルは揃っているか？
- [ ] ユーザー管理に必要なテーブルは？
- [ ] 組織/グループ管理に必要なテーブルは？
- [ ] コア機能に必要なテーブルは？
- [ ] 外部連携に必要なテーブルは？

## データフロー視点
- [ ] ユーザー登録フローで必要なデータは？
- [ ] 認証フローで必要なデータは？
- [ ] 主要ビジネスフローで必要なデータは？

## ライフサイクル視点
- [ ] 削除時の処理（論理削除/物理削除）は定義済み？
- [ ] アーカイブ要件は考慮済み？
- [ ] データ保持期間は定義済み？

## 関係者視点
- [ ] 管理者（admin）用のデータは？
- [ ] 一般ユーザー用のデータは？
- [ ] システム連携用のデータは？

## 非機能視点
- [ ] 監査ログ用のテーブルは？
- [ ] メトリクス・統計用のテーブルは？
- [ ] 通知・配信用のテーブルは？
```

### 5.4 MASTER.mdの役割

MASTER.mdは全ドキュメントの**中央ハブ**として機能:

```markdown
## MASTER.mdに必須の内容

1. プロジェクト識別情報
   - プロジェクト名、バージョン、最終更新日

2. 30秒概要
   - 何をするプロジェクトか一言で

3. 技術スタック
   - 使用技術の一覧

4. コード生成ルール
   - 型安全性、マジックナンバー禁止など

5. 命名規則
   - 変数、関数、クラス、ファイルの命名

6. Git Workflow
   - ブランチ戦略、コミット規約

7. ドキュメント索引
   - 7つの必須ドキュメントへのリンク
```

## 6. Phase 5: Git Workflow

### 6.1 ブランチ戦略

```
┌─────────────────────────────────────────────────────────────────┐
│                    ブランチ戦略                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  main (本番)                                                     │
│    ↑ マージ（リリース時のみ）                                     │
│  develop (開発ベース)                                            │
│    ↑ PRマージ                                                    │
│  feature/#123-add-auth (機能開発)                                │
│                                                                 │
│  ブランチ命名規則:                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ feature/#{issue-number}-{description}  # 新機能          │   │
│  │ fix/#{issue-number}-{description}      # バグ修正        │   │
│  │ chore/#{issue-number}-{description}    # 雑務            │   │
│  │ docs/#{issue-number}-{description}     # ドキュメント     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 コミットメッセージ規約

```
<type>: #<issue> <subject>

<body>

Closes #<issue>
```

#### Type一覧

| Type | 用途 | 例 |
|------|------|-----|
| `feat` | 新機能 | `feat: #3 認証APIを実装` |
| `fix` | バグ修正 | `fix: #45 ログイン失敗時のエラー修正` |
| `docs` | ドキュメント | `docs: #10 ARCHITECTURE.md更新` |
| `style` | フォーマット | `style: コードフォーマット適用` |
| `refactor` | リファクタリング | `refactor: 認証ロジックを分離` |
| `test` | テスト | `test: #3 認証APIのテスト追加` |
| `chore` | 雑務 | `chore: 依存関係更新` |

### 6.3 PRのセルフレビュー

```markdown
## PR作成前のセルフレビューチェックリスト

### DRY原則
- [ ] 重複コードはないか？
- [ ] 重複インポートはないか？
- [ ] マジックナンバーは定数化されているか？

### コード品質
- [ ] 型アノテーションは適切か？
- [ ] エラーハンドリングは適切か？
- [ ] 変数名・関数名は意味が明確か？
- [ ] デバッグログは削除したか？

### インポート整理
- [ ] stdlib → third-party → local の順か？
- [ ] 未使用インポートはないか？

### ドキュメント
- [ ] 必要なdocstringは書いたか？
- [ ] 複雑なロジックにコメントはあるか？

### テスト
- [ ] 新規コードにテストはあるか？
- [ ] 既存テストは更新したか？
- [ ] 全テストがパスするか？
```

## 8. アンチパターン

### 8.1 避けるべきパターン

| アンチパターン | 問題点 | 正しいアプローチ |
|--------------|--------|----------------|
| **Issueなし開発** | 要件が不明確、追跡不可 | Issue-First |
| **口頭合意のみ** | 後で忘れる、認識齟齬 | ドキュメントに明記 |
| **一気に全部設計** | 漏れが発生しやすい | MECE検証で段階的に |
| **ドキュメント分散** | 情報が見つからない | MASTER.mdで一元管理 |
| **適当なコミット** | 履歴が追えない | 規約に従ったコミット |

### 8.2 よくある失敗と対策

```
❌ 失敗: DBスキーマを後から大幅変更
✅ 対策: Phase 3のMECE検証を徹底

❌ 失敗: 実装時にルールがバラバラ
✅ 対策: MASTER.mdでルールを明文化

❌ 失敗: PRが大きすぎてレビュー困難
✅ 対策: Issueを適切な粒度に分割

❌ 失敗: ドキュメントが古くなる
✅ 対策: 変更時に必ずドキュメント更新（Revision History）
```
